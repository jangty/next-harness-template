# 요구사항 발견 (Requirements Discovery)

제품 아이디어를 **검증 가능한 명세**로 바꾼다. 코드 전에 *무엇을 · 왜*를 확정해 바이브코딩의 혼란을 막는다.

> 전체 단계 흐름은 `/kickoff` 명령에 있다. 사용자가 `/kickoff` 없이 아이디어를 설명하기 시작하면, 이 방법으로 그 흐름을 시작한다.
> ⚠️ 빈칸은 추측하지 않는다(§andrej-karpathy 1: 가정 명시, 불확실하면 묻는다).

## 인터뷰 방법
- 질문을 **묶음으로** 던진다 — 한 번에 하나씩 묻지 말 것(왕복 낭비). 답을 받아 명세에 반영하고 빈 곳만 다시 묻는다.
- 소크라테스식: "왜", "그게 안 되면 어떻게 되나", "어떻게 측정하나"로 모호함을 깎는다.
- 사용자가 해법부터 말하면 **문제로 되돌린다** — "그게 해결하려는 진짜 일(JTBD)은?"

## 작성 컨벤션 (산출물 형식)

**기능요건 — 유저스토리 + 수용기준**
- 스토리: `<역할>로서 <목표>를 원한다, 그래야 <가치>.`
- 수용기준(Given/When/Then): `Given <상황>, When <행동>, Then <기대 결과>.`
- 각 Must-have 스토리에는 최소 1개의 수용기준이 붙어야 한다(테스트로 직결 → `docs/verification.md`).

**시스템 수준 요건 — EARS (선택)**
- 유비쿼터스: `시스템은 <응답>해야 한다.`
- 이벤트: `WHEN <트리거>, 시스템은 <응답>해야 한다.`
- 상태: `WHILE <상태>, 시스템은 <응답>해야 한다.`
- 조건: `IF <조건>, THEN 시스템은 <응답>해야 한다.`
- 선택: `WHERE <기능 포함>, 시스템은 <응답>해야 한다.`

**비기능요건(NFR) — 빠뜨리기 쉬움, 명시적으로**
- 성능·확장(목표 부하/응답시간) · 보안·프라이버시(개인정보·인증·인가 → `docs/security.md` ADR) · 가용성 · 접근성(a11y) · 관측성 · 규정/컴플라이언스.

**우선순위 — MoSCoW**
- Must(MVP 필수) / Should / Could / Won't(이번엔 안 함 = 비목표). MVP는 Must의 가장 얇은 슬라이스.

## Definition of Ready (코딩 진입 게이트)
명세가 이걸 만족해야 스캐폴딩으로 넘어간다: 핵심 절에 TODO 없음 · 모든 Must에 수용기준 · 성공지표 측정 가능 · 비목표 명시 · MVP 정의 · 함의된 결정은 ADR 스텁.

## 기록 시스템 연결
산출물은 `docs/product/{brief,requirements}.md`. 거기서 ADR(왜)·exec-plan(계획)·HANDOFF(연속성)·CLAUDE.md(불변식)로 흘려보낸다. 함의된 아키텍처 결정은 미리 정하지 말고 ADR 스텁으로 남긴다(템플릿 규칙).
